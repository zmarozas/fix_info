1 · Executive summary

The MOS transaction cache faces unsustainable heap growth — over 32GB at peak with 1M+ intraday orders — requiring daily restart to operate. The Prague team's March 2024 V2 design evaluated three approaches (Kafka-based, local file/DB, application separation) but reached no decision. The page has been dormant since then.

This proposal recommends a tiered architecture combining Neeve (write path) with Hazelcast (distributed query store). Hazelcast is the preferred technology because it is verifiably in production at Barclays today (USCB COO team's March 2026 deployment), with established internal expertise and bank-scale trading case studies (Morgan Stanley, HSBC, JP Morgan/Highbridge Capital).

Bottom line
Keep Neeve. Keep Solace. Keep .rdat. Add Hazelcast as a distributed query layer fed by the Neeve TC primary. Evict terminal orders from Neeve heap; Hazelcast retains them for Batman/GUI queries through the existing message-driven interface. Zero changes to external consumer apps.

2 · Problem statement

The Transaction Cache (TC) operates as a single Neeve agent with active backup, holding all order state directly in Java Maps. As the OMS publishes order events through Solace, the TC builds a materialised view that serves queries from external apps including GUI clients, Batman, and the dfd-command tool.

Current responsibilities (per Prague design)

  • Provide the latest version of orders to Batman and other clients (intraday restarts)
  • Provide trade and order history to Batman for the historical blotter view
  • Provide GTCs, GTDs and other long-lived orders to the Blackbird OMS, since the OMS keeps order objects only for a day or week depending on its restart schedule

Constraints driving the redesign

  • Heap exhaustion: EOD heap reaches extreme size, daily restart required to operate
  • Single-node ceiling: Primary/backup mirrors full state — memory cost doubled across topology
  • External app coupling: Batman, GUI, OMS depend on existing message-driven interface — cannot change APIs
  • Mixed query patterns: Hot intraday reads + full-day cross-symbol queries + GTC/GTD overnight queries
  • Existing infrastructure investment: Neeve .rdat journal, Solace fabric, OMS sharding model — all working well, must be preserved

4 · Design pitfalls in the EU proposal

The Prague team's analysis is rigorous, but each of their three approaches has structural problems that justify proposing an alternative direction. The pitfalls below are direct from their own documented Cons section, plus additional concerns identified through this analysis.

PITFALL 01 · Kafka approach
Slow restoration on restart — replay from beginning
With 1M+ orders accumulated through a trading session, replaying every event from a Kafka topic on application restart takes minutes. During this window the TC is unavailable. The Prague team explicitly acknowledges this is a major drawback.
"Long restoration — Messages must be replayed from the beginning to achieve the same state."
PITFALL 02 · Kafka approach
Data duplicated in Kafka cluster AND application heap
Kafka becomes an additional storage tier — the entire dataset exists in Kafka topics, in primary heap, and in backup heap simultaneously. This compounds rather than solves the original heap problem. Storage cost is now 3× the dataset size.
"Data will be duplicated, in Kafka cluster and in application nodes — Kafka is used only as message broker for failure restoration."
PITFALL 03 · Kafka approach
Application start needs to read every message
The cons section explicitly notes that startup reads the entire topic. This makes intraday restart practically impossible — a primary failure during market hours becomes a multi-minute outage. For a trading system this is operationally unacceptable.
"Application start needs to read every message."
PITFALL 04 · Kafka approach
Intraday resharding is "complicated" — i.e. unsolved
The notes section acknowledges that adding shards during the trading day is theoretically possible but practically complicated. This means the design cannot respond to volume spikes, and capacity planning becomes a one-time-per-day exercise.
"In theory ability to do intraday resharding — adding more nodes on high demand — however it will be complicated."
PITFALL 05 · Local file / embedded DB approach
Cannot add new shards during the day
The embedded DB option ties storage to each node's local disk. New nodes have no historical data and cannot catch up because the feed isn't shared. This is acknowledged as "almost impossible" in the design notes.
"Intraday adding more nodes will be almost impossible as the feed will not be in sync (for embedded DB only)."
PITFALL 06 · Local file / embedded DB approach
Storing data as opaque blobs prevents queries
The embedded DB cons explicitly mention "storing data as a blob" and "filtering" as cons. This means cross-symbol queries (Batman's account-level view) require full scans, which scales poorly at 1M orders.
"Cons: Filtering. Storing data as a blob. The overall loss of ability to replicate the flow."
PITFALL 07 · Synchronized DB approach
Failover causes data duplication inside the database
With external DB replication, when primary fails and backup takes over, in-flight writes can land twice. The Prague team flags that "proper queries" are needed to handle this, but the design doesn't specify them.
"Update of data when switch happened can lead to duplication of data inside DB → need to have proper queries."
PITFALL 08 · Application separation approach
Trader search transferred to external project — loss of control
Splitting the cache means handing off the trader query path to another team's project (TC2). This adds organisational dependency and removes our team's ability to optimise for our specific access patterns.
"Trader search would be transferred to TC2 project → data no longer under our control."
PITFALL 09 · Application separation approach
Regional rollout uncertainty
The Prague team notes uncertainty about whether the split can be done for one region only. For a global trading platform, asymmetric architectures across regions create operational complexity and add risk to deployment sequencing.
"Not sure if this can be done for one region only."
PITFALL 10 · Common to all three approaches
Distributed cache option was never seriously evaluated
The 2024 design considered Kafka, local files, and application separation. It did not evaluate distributed in-memory data grids (Hazelcast, Infinispan, Coherence) as a primary storage option. My August 2024 comment raised this gap; it received no response. With 20 months of organisational evolution and concrete Hazelcast adoption at Barclays since then, this gap can now be closed.
Net assessment of EU proposal
The Prague design is technically thorough but each approach trades one problem for another. Kafka solves durability but creates 3× storage and slow restart. Local file solves restart but breaks resharding. Application separation solves heap but loses control. None of the three solves the core problem cleanly. A distributed cache approach addresses all three pain points simultaneously, which is why it warrants formal proposal now.

7 · Recommended approach: Hazelcast tiered architecture

Adopt a tiered storage model where Neeve TC retains the active write path with intelligent eviction, and Hazelcast holds the full intraday dataset for query access. The existing message-driven query interface (used by Batman, GUI, OMS) is preserved entirely — Hazelcast is a TC-internal implementation detail.

Why Hazelcast over the EU proposals

Kafka-based (EU proposal)
Slow restart, 3× data duplication, complicated resharding, no native query support. Acknowledged in EU's own cons.
Hazelcast (this proposal)
Fast restart (data already in cluster), native indexed queries, partition rebalance built in, single source of truth.
Embedded DB (EU proposal)
Cannot add nodes intraday. Blob storage prevents cross-symbol queries. Restoration depends on local disk integrity.
Hazelcast (this proposal)
Add cluster nodes any time. Indexed by accountId, deskId, symbol, status. Distributed redundancy by design.
Application separation (EU proposal)
Hand off trader query path to TC2 project. Loss of control. Regional rollout uncertain.
Hazelcast (this proposal)
Internal architectural change. No project boundaries crossed. Same team retains ownership.

Why Hazelcast over Infinispan

  • Verified Barclays adoption: Production deployment in USCB (March 2026)
  • Internal expertise: Operational knowledge already exists in the firm
  • Trading-bank case studies: Morgan Stanley, HSBC, JP Morgan/Highbridge
  • Learning ecosystem: More books, courses, community content available
  • Off-heap "Elastic Memory": Mature feature for trading workloads (avoids GC pressure)
  • Lower procurement friction: Existing relationship vs. negotiating a new vendor

Infinispan remains a strong technical alternative — see section 12 for full comparison. The recommendation is decision-conditional: if internal verification reveals an active Red Hat Data Grid deployment, Infinispan should be re-evaluated as a peer option.

9 · Message-driven query interface

Critical compatibility constraint: Batman and GUI clients do not use a library. They send query messages to Solace topics and receive response streams (header followed by event stream), the same pattern used by OMS event replay. The Hazelcast cluster is therefore a TC-internal implementation detail. External clients see zero API changes.

Query flow: Batman Solace TC Primary Hazelcast │ │ │ │ │── OrderQueryRequest ──▶ │ │ │ │── deliver to TC ────────▶│ │ │ │ │── route by scope ─────▶│ │ │ │ (cross-symbol) │ │ │ │◀── result list ────────│ │ │ │ │ │ │◀── ResponseHeader (count)│ │ │◀──────────────────────│ │ │ │ │◀── OrderEvent stream ────│ │ │◀──────────────────────│ │ │ │ │◀── ResponseFooter ───────│ │ │◀──────────────────────│ │ │

Query routing logic

// QueryRequestHandler.onQueryRequest(OrderQueryRequest req)

QueryScope scope = determineScope(req);

switch (scope) {
    case OVERNIGHT:
        // GTC/GTD lookup — local Neeve overnight map (~1000 orders)
        results = overnightOrders.values().stream()
            .filter(matchTimeInForce(req))
            .filter(matchSymbols(req))
            .toList();
        break;
        
    case ACTIVE:
        // Working order lookup — local Neeve active map (~50-200k)
        results = activeOrders.values().stream()
            .filter(matchSymbols(req))
            .filter(matchDesk(req))
            .toList();
        break;
        
    case FULL_DAY:
        // Cross-symbol or all-state query — Hazelcast IMap query
        Predicate p = Predicates.and(
            Predicates.equal("deskId", req.getDeskId()),
            Predicates.in("symbol", req.getSymbols())
        );
        results = ordersIMap.values(p);
        break;
}

// Stream response back through Solace (existing pattern)
publisher.send(replyTopic, new ResponseHeader(req.getId(), results.size()));
results.forEach(o -> publisher.send(replyTopic, toEvent(o)));
publisher.send(replyTopic, new ResponseFooter(req.getId()));

10 · Tiered memory model

Tier 1: Neeve activeOrders (Java Map, on-heap) └── Working orders only (NEW, WORKING, PARTIALLY_FILLED) └── Evicted on terminal state (FILLED, CANCELLED, REJECTED, EXPIRED) └── Evicted on idle timeout └── Evicted on memory watermark (LRU safety net) └── Size: ~50-200k orders ≈ 2-4 GB heap └── Mirrored on Backup via Neeve native HA Tier 2: Neeve overnightOrders (Java Map, on-heap) └── GTC, GTD, day+ orders └── Loaded from JSON flat file at startup └── Stays local (small ~1000 orders) └── Persisted at EOD to flat file └── Mirrored on Backup Tier 3: Hazelcast IMap "todays-orders" (off-heap, distributed) └── ALL of today's orders regardless of state └── Populated by Neeve Primary on every order event └── Indexed: accountId, deskId, symbol, status, TIF └── Cleared at EOD └── Backup count: 1 (primary + backup partition per node) └── Size: full 1M+ orders, distributed across cluster Tier 4: Neeve .rdat journal (append-only, durable) └── Audit trail of every event └── Used for disaster recovery only └── Not used in normal startup (flat file is) Tier 5: Overnight flat file (JSON, ~1000 orders) └── Carry-over to next trading day (unchanged from today)

Eviction rules (combined)

  1. Terminal state: Filled, Cancelled, Rejected, Expired → evict immediately after final event written
  2. Idle timeout: No activity for N minutes (configurable) → evict
  3. Memory pressure: If activeOrders.size() exceeds high watermark → evict oldest (LRU safety net)
Atomicity rule
Always confirm Hazelcast write succeeded BEFORE removing from local Neeve map. Briefly in both = acceptable (same data). Never in neither = unacceptable. The eviction handler uses synchronous put for terminal-state writes; async put for in-flight updates.

11 · Failure & recovery scenarios

SCENARIO 01 · Normal operation Steady state

OMS publishes order events. TC Primary consumes, updates activeOrders, writes to Hazelcast. TC Backup consumes same events, mirrors local state, no Hazelcast writes. Batman queries flow through Solace request/response, served from local maps or Hazelcast based on scope. .rdat journals every event on both nodes.

SCENARIO 02 · TC Primary crashes HA event
T0 + 0s
Primary failure detected
Neeve cluster manager detects heartbeat loss. Backup begins promotion. Hazelcast unaffected — its cluster is independent.
T0 + ~3s
Backup promotes to Primary
Local activeOrders map already current (mirrored). New Primary begins writing to Hazelcast. Brief async window: in-flight Hazelcast writes from old Primary may have been lost.
T0 + ~5s
Reconciliation
New Primary iterates its activeOrders map, confirms each order is in Hazelcast, re-writes any missing. Bounded time (seconds for 50-200k orders).
T0 + ~10s
Service resumes
OMS event flow continues. Batman queries continue (Hazelcast was never affected). System operates with no Backup until restoration.
SCENARIO 03 · Restoring failed node as new Backup Recovery procedure
Phase A · ~2 min
Boot & isolate
Provision/restart host. Start Neeve TC standalone — NOT registered as backup yet. Verify connectivity to Solace, Hazelcast, peer Primary. Hazelcast writes disabled.
Phase B · 1-10 min
Replay .rdat + Solace catch-up
Replay local .rdat with eviction logic applied inline (so heap doesn't explode). Subscribe to Solace from appropriate offset. Process events through normal handlers. Continue until lag < 100ms.
Phase C · ~1 min
Register as Backup
Register with Neeve cluster. Verify mirroring active. Run optional state checksum vs. Primary.
Phase D · ~2 min
Validate HA
Send synthetic order. Verify both P&B receive, only Primary writes to Hazelcast. Confirm Lumino dashboard healthy. Sign off.

Total: ~15 minutes typical, up to 45 worst case. No failback — the promoted Backup remains Primary; restored node becomes Backup.

SCENARIO 04 · Both Neeve TC nodes fail Disaster recovery

Total TC outage. However, Hazelcast cluster remains operational. Batman/GUI queries continue to be served (degraded — no live event updates, but historical view intact). Recovery: provision Neeve hosts, replay .rdat from latest snapshot. Hazelcast can be used as a fast-recovery source for Neeve activeOrders reload (read all entries back into local map at boot) — eliminates the slow Kafka-replay problem the EU proposal had.

SCENARIO 05 · Hazelcast node failure Self-healing

Hazelcast cluster has backup partitions on different nodes (numBackups=1). Failed node's data is automatically promoted on its backup-holding peer. Cluster rebalances. Neeve TC continues operating — Hot Rod / smart client routes around the failed node automatically. No human intervention required.

SCENARIO 06 · Total Hazelcast cluster failure Rare but possible

Batman/GUI queries that need cross-symbol or full-day view fail. Neeve TC continues operating (writes & .rdat unaffected). Recovery path: rebuild Hazelcast cluster, then have Neeve Primary scan its activeOrders + replay recent .rdat events to repopulate. Estimated time: minutes for partial rebuild, longer for full historical reconstruction.

SCENARIO 07 · Solace outage Platform-level

Out of TC scope — Solace is a platform-wide dependency. Trading halts firm-wide. Neeve TC has nothing to consume; Hazelcast retains last known state. Recovery is platform team responsibility.

SCENARIO 08 · End-of-day Daily routine

17:00 EOD process: identify still-open orders (GTC/GTD/working), write to overnight flat file. Clear Hazelcast cache (cache.clear() all nodes). Flush Neeve activeOrders. Roll .rdat. System idle. Next morning 07:00: Neeve reads overnight flat file → populates local maps → Primary writes to Hazelcast → ready for market open.

SCENARIO 09 · Intraday volume spike Capacity scaling

Unique advantage over EU proposals: add Hazelcast nodes during the trading day. Hazelcast rebalances partitions automatically. No restart, no resharding ceremony. Direct response to one of the EU design's "complicated" weaknesses.

Closing statement
The Prague team's 2024 V2 design identified the right problem but each of their three approaches trades one weakness for another. None solves heap, restart, and resharding cleanly. A distributed in-memory data grid does — and Hazelcast is the strongest candidate at Barclays specifically, given verified 2026 deployment in USCB, established trading-bank case studies at peer institutions, and an existing licensing relationship.

This proposal preserves every working component (Neeve, Solace, .rdat, message-driven query interface) and adds one strategically-aligned component (Hazelcast) to absorb the heap and query growth. External app compatibility is preserved entirely. The architecture is conservative, defensible, and aligned with the firm's actual technology direction.
