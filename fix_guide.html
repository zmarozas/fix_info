Practical FIX Guide for Backend Trading Developers

1. Why FIX matters for us

FIX is the common language between our systems (OMS/EMS) and external brokers/venues. It carries every important event in an order’s life: submission, acknowledgments, fills, cancels, rejects.

If you understand FIX:

  • You can trace an order end‑to‑end across systems.
  • You can debug production issues from logs instead of guessing.
  • You can design APIs and services that align with how markets actually behave.

This guide is intentionally practical and short. It focuses on the subset of FIX that backend developers touch daily.

2. FIX fundamentals

2.1 What FIX is

  • A text-based message protocol for trading-related workflows (orders, executions, allocations, etc.). See the FIX Online Specification: https://www.fixtrading.org/online-specification/
  • Messages are tag=value pairs separated by delimiters (SOH on the wire, often shown as | in logs).
  • Each message has:
    • Standard Header: BeginString(8), BodyLength(9), MsgType(35), SenderCompID(49), TargetCompID(56), MsgSeqNum(34), SendingTime(52).
    • Body: business fields (e.g., ClOrdID(11), Symbol(55), Side(54), OrderQty(38), Price(44)).
    • Trailer: CheckSum(10).
  • Good “anatomy” reference: OnixS message anatomy: https://www.onixs.biz/insights/the-anatomy-of-fix-protocol-messages

2.2 Sessions vs application messages

  • Session layer: connection, logon, heartbeats, sequence numbers, recovery, logout.
    • FIX Session Layer Online: https://www.fixtrading.org/standards/fix-session-layer-online/
  • Application layer: orders, executions, cancels, rejects (NewOrderSingle, ExecutionReport, etc.).
    • FIX Online Specification (business layer entry): https://www.fixtrading.org/online-specification/

Mental model:

  • Session layer answers: “Are we connected and in sync?”
  • Application layer answers: “What happened to this order?”

2.3 Key message types (by role)

Admin

  • Logon (35=A) – start a FIX connection.
  • Heartbeat (35=0) – keepalive when idle.
  • TestRequest (35=1) – “are you alive?”.
  • ResendRequest (35=2) – “I missed messages; resend from X to Y”.
  • Logout (35=5) – end of session.

Application

  • NewOrderSingle (35=D) – submit a new order.
    • Example dictionary: FIX 5.0 SP2: https://www.onixs.biz/fix-dictionary/5.0.sp2/msgtype_d_68.html
  • OrderCancelRequest (35=F) – request cancel.
  • OrderCancelReplaceRequest (35=G) – amend order (quantity/price).
  • ExecutionReport (35=8) – ALL order events (ack, partial fill, fill, cancel, reject…).
    • FIX 4.2: https://www.onixs.biz/fix-dictionary/4.2/msgtype_8_8.html
    • FIX 5.0 SP2: https://www.onixs.biz/fix-dictionary/5.0.sp2/msgtype_8_8.html
  • OrderCancelReject (35=9) – cancel/replace rejected.

3. Order lifecycle in FIX

3.1 Core fields for order identity and quantities

  • ClOrdID(11): client-assigned order ID (our OMS ID).
  • OrderID(37): broker/exchange-assigned ID.
  • OrigClOrdID(41): links cancels/replaces to the original order.
  • Symbol(55), Side(54), OrderQty(38), Price(44).
  • CumQty(14): total executed quantity.
  • LeavesQty(151): remaining quantity.
  • LastQty(32), LastPx(31): size and price of the most recent fill.

3.2 ExecutionReport: the workhorse

Every state change comes as an ExecutionReport (35=8) with:

  • ExecType(150): what happened (NEW, PARTIAL_FILL, FILL, CANCELED, REJECTED, REPLACED, etc.).
  • OrdStatus(39): current order state.

Useful state/transition docs:

  • Order State Changes: https://www.fixtrading.org/online-specification/order-state-changes/

Typical OrdStatus(39) values:

  • 0 – New
  • 1 – Partially filled
  • 2 – Filled
  • 4 – Canceled
  • 8 – Rejected
  • 6 – Pending Cancel
  • E – Pending Replace

3.3 Example lifecycle

Scenario: Limit order for 10,000 shares.

  1. Client → Broker: NewOrderSingle (35=D) with ClOrdID=ABC1, OrderQty=10000, Price=10.00.
  2. Broker → Client: ExecutionReport NEW (ExecType=0, OrdStatus=0, CumQty=0, LeavesQty=10000).
  3. Partial fill: ExecutionReport PARTIAL_FILL (ExecType=1, OrdStatus=1, LastQty=2000, CumQty=2000, LeavesQty=8000).
  4. Final fill: ExecutionReport FILL (ExecType=2, OrdStatus=2, LastQty=8000, CumQty=10000, LeavesQty=0).

How devs use this:

  • When debugging “order stuck” issues, check the sequence of ExecutionReports and verify CumQty/LeavesQty and OrdStatus progression.
  • If the UI shows Filled but the last ExecutionReport has OrdStatus=1, there’s a state-mapping bug.

4. Session management

4.1 Connection and logon

  • FIX rides on a TCP socket; once connected, Logon (35=A) must be the first message from the initiator.
  • Key Logon fields:
    • SenderCompID(49), TargetCompID(56) – identities.
    • HeartBtInt(108) – heartbeat interval (seconds).
    • MsgSeqNum(34) – session sequence number.
    • Optional ResetSeqNumFlag(141) – sequence reset behavior.
  • Spec: https://www.fixtrading.org/standards/fix-session-layer-online/

Rule of thumb: No application messages (orders) should be sent before you receive the peer’s Logon ack.

4.2 Heartbeats and liveness

  • During idle periods, both sides send Heartbeat (35=0) at HeartBtInt intervals.
  • If you suspect the peer is stuck, send TestRequest (35=1); peer must respond with Heartbeat including TestReqID(112).
  • No heartbeat / response within agreed timeout → treat as session down.

4.3 Sequence numbers and recovery

  • MsgSeqNum(34) increments by 1 for each message in a session.
  • If a party detects a gap (e.g., sees seq 1, 2, 5), it sends ResendRequest (35=2) asking for missing messages.
  • Peer responds with:
    • Resent messages with PossDupFlag(43)=Y, and/or
    • Gap fills (SequenceReset with GapFillFlag=Y) when missing messages are intentionally skipped.
  • Session test cases: https://www.fixtrading.org/standards/fix-session-testcases-online/

For developers: The FIX engine should hide most of this, but you need it to interpret logs and avoid double-processing duplicates.

5. Common failure scenarios

5.1 Disconnects

Symptoms:

  • No heartbeats, socket closed, or repeated TestRequests.
  • Orders may be in-flight; last state in our OMS may not match the venue.

What to look for:

  • Last few admin messages (TestRequest, Heartbeat, Logout).
  • Whether sequence numbers were in-sync before disconnect.
  • Any ResendRequests on reconnection.

5.2 Duplicates

Cause:

  • Session recovery/resend after disconnect; peer resends older messages with PossDupFlag(43)=Y.

Implications:

  • Order/position services must treat ExecutionReports with the same ExecID(17) as idempotent.
  • State updates should be robust to reprocessing old fills.

5.3 Out-of-sync sequences

Symptoms:

  • Logon rejected due to unexpected MsgSeqNum.
  • Immediate ResendRequest after logon.

Actions:

  • Check which side is ahead/behind.
  • Confirm message store configuration (where historical messages are loaded from).
  • Ensure you’re not resetting sequences without mutual agreement.

5.4 Business rejects

Examples:

  • OrderCancelReject (35=9).
  • ExecutionReport with OrdStatus=8 (Rejected) and Text(58) explaining the business reason.

Typical causes:

  • Invalid price/quantity.
  • Order too late to cancel.
  • Instrument not tradable in that venue/session.

How to debug:

  • Look at Text(58) and OrdRejReason(103).
  • Compare our request fields against the venue’s rules (price increments, min size, etc.).

6. Practical system perspective (OMS/EMS/exchange)

6.1 Typical architecture

  • OMS (Order Management System):
    • Owns client-side order IDs (ClOrdID).
    • Applies internal rules (risk, credit, compliance).
    • Routes to brokers/venues via FIX or other APIs.
  • EMS (Execution Management System):
    • Closer to execution, multi-venue routing, algorithms.
    • Translates high-level instructions into venue-specific FIX flows.
  • Exchange/Broker gateway:
    • FIX acceptor.
    • Validates session, business rules, routes to matching engine or external venues.

6.2 How a single order flows

  1. User/trader submits order via UI → OMS.
  2. OMS validates, assigns ClOrdID, persists state.
  3. OMS → Broker via NewOrderSingle (35=D).
  4. Broker sends ExecutionReports: NEW → PARTIAL_FILL → FILL/CANCELED.
  5. OMS updates internal state and downstream services based on ExecutionReports.

Key point: the source of truth for execution state is the sequence of ExecutionReports from the venue. Our internal state should be a faithful projection of that.

7. FIX for debugging and production support

7.1 Reading logs

Typical production incident questions: “my order disappeared”, “partial fill not reflected”, “cancel too late”. Start with:

  • FIX session logs:
    • Filter by CompIDs and ClOrdID(11).
    • Confirm the chain: Logon → NewOrderSingle → ExecutionReports → Logout.
  • Application logs:
    • Where we parse FIX and map to internal entities.

7.2 Typical debug questions

  • Did we actually send the NewOrderSingle? (Check ClOrdID and MsgSeqNum in outbound logs.)
  • Did the broker acknowledge it as NEW? (ExecutionReport with ExecType=0, OrdStatus=0.)
  • Were there any ResendRequests that produced duplicates or reordering?
  • Is our view of CumQty/LeavesQty consistent with the last ExecutionReport?

7.3 Building better tooling

  • “Order timeline” view that reconstructs state from raw FIX logs.
  • Alerting when:
    • Session is up but no heartbeats.
    • High rate of ResendRequests.
    • Repeated OrderCancelRejects for the same order.
