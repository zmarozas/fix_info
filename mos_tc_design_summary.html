1 · Problem statement

The OMS uses Neeve and is sharded by symbol. The Transaction Cache (TC) also uses Neeve but runs as a single primary with a hot-standby backup. The TC stores order events as Java Maps — the latest copy of all order state — and serves queries from external apps including GUI, Batman, and overnight order tooling.

Scale & pressure

  • Intraday volume: 1M+ orders during peak sessions
  • Heap pressure: Transaction cache heap can exceed 32GB
  • Overnight: ~1000 unfilled orders carried via flat-file JSON
  • Replication: Full state replication between primary & backup (memory cost doubled)
  • Durability: Neeve .rdat journal records every event for crash recovery
  • Messaging: Solace fabric (Neeve's exactly-once transport)
The brief: Boss asked to think about the design — this is an architectural exercise, not a fire drill. Goal is to bound TC heap and provide horizontal scalability without breaking the many external apps that read from it.

2 · Why not shard Neeve TC the same way as OMS

The instinct to mirror OMS sharding (consistent hash on symbol) is the wrong solution for TC because the two components serve different purposes:

AspectOMSTransaction Cache
Primary purpose Order processing & lifecycle management Materialized read model fed by OMS events
Write pattern Per-symbol — natural shard boundary Receives events, stores latest state
Read pattern Internal to OMS External apps query: by account, desk, TIF, full-day
Sharding by symbol Natural fit Breaks consumers — most queries are not symbol-keyed
Conclusion: TC is essentially a CQRS read-model / materialized view. Sharding the read side by the same key as the write side optimises the wrong thing — it punishes the actual usage pattern (cross-symbol queries by Batman/GUI) to fix a heap problem.

3 · Tiered memory architecture (the chosen approach)

Instead of sharding Neeve, split the TC's responsibilities across tiers. Neeve stays lean and handles the hot write path. A distributed in-memory data grid absorbs the query load and the unbounded state growth.

Tier 1: Neeve activeOrders (Java Map) └── Working orders only — small, hot └── Evict on terminal state OR memory pressure └── Primary + Backup mirror Tier 2: Neeve overnightOrders (Java Map) └── ~1000 GTC/GTD orders carried overnight └── Loaded from flat file at startup └── Stays local (small enough to be cheap) Tier 3: Distributed Data Grid (Infinispan) └── ALL of today's orders regardless of state └── Populated by Primary on every event └── Evicted orders from Tier 1 stay queryable here └── Cleared at EOD Tier 4: Neeve .rdat journal └── Append-only, audit + disaster recovery └── Not used in normal operation Tier 5: Overnight flat file └── ~1000 orders carried to next day └── Unchanged from today's pattern
Heap math: Today's Neeve TC heap = 32GB+ holding all 1M orders. Proposed: Neeve heap = 2-4GB (active working set only). The full 1M dataset moves to Infinispan, partitioned across cluster nodes — no single JVM heap problem.

Layer responsibilities

  • OMS publishes order events to Solace. Unchanged.
  • Solace delivers events to TC and carries query request/response messages. Unchanged.
  • TC Primary consumes events, updates local state, mirrors to Infinispan, evicts terminal orders.
  • TC Backup consumes same events, maintains shadow state, does not write to Infinispan.
  • Infinispan holds full-day state, serves queries via the TC's QueryRequestHandler.

6 · Message-driven query interface

Critical clarification: Batman and GUI clients do NOT use a library. They send query messages to Solace topics and receive a response stream — header message followed by a stream of order events, similar to OMS event replay. This is the same pattern they use today.

Why this matters: Infinispan is a TC-internal implementation detail. No external client ever queries Infinispan directly. The TC exposes a message-based query interface that internally routes to the right tier (local active map, local overnight map, or Infinispan grid). External apps see zero API change.

Query flow

  1. Client (Batman/GUI) publishes OrderQueryRequest to trading/tc/query/request
  2. TC QueryRequestHandler receives the request
  3. Handler determines query scope (active / overnight / full-today) based on filter criteria (symbols, time-in-force, desk, etc.)
  4. Handler routes to appropriate tier:
    • Active orders → local Neeve activeOrders map
    • Overnight (GTC/GTD) → local Neeve overnightOrders map
    • Full-day / cross-symbol → Infinispan via Hot Rod query
  5. Handler publishes response header (count, request ID) to trading/tc/query/response/{requestId}
  6. Handler streams individual order event messages on same response topic
  7. Optional footer message marks end of stream

8 · Protobuf serialization alignment

Existing Solace messages and OMS events are protobuf-encoded. Infinispan uses ProtoStream marshaller natively — same wire format. This is a meaningful alignment that strengthens the Infinispan choice.

Benefits of end-to-end protobuf

  • Single schema source of truth: shared order.proto across Solace messages, TC internal model, Infinispan storage, and query response messages
  • No format translation overhead: bytes flow consistently through the pipeline
  • Indexable inside Infinispan: ProtoStream allows querying on protobuf fields directly — Infinispan can index accountId, symbol, deskId, status inside protobuf-encoded values
  • Schema evolution: protobuf's backward/forward compatibility rules apply uniformly across the stack
  • Wire format compatibility: ProtoStream and Google protobuf-java produce identical bytes — no interop concern

9 · Eviction logic

The key behavioural change: TC primary evicts orders from its local activeOrders map once they reach terminal state. Data remains queryable via Infinispan.

Eviction rules (combined)

  • Terminal state: Filled, Cancelled, Rejected, Expired → evict immediately
  • Idle timeout: No activity for N minutes → evict (configurable)
  • Memory pressure: If activeOrders.size() exceeds high watermark → evict oldest (LRU backup safety net)

Eviction safety rule

Atomic-ish eviction: Always confirm Infinispan write succeeded BEFORE removing from local map. Briefly in both = fine (same data). Never in neither = unacceptable. 
public void evict(Order order) {
    if (nodeRole.isPrimary()) {
        boolean confirmed = infinispanWriter.putAndConfirm(order);
        if (!confirmed) {
            // don't evict — keep in local map, alert ops
            return;
        }
    }
    activeOrders.remove(order.getOrderId());
}

Replay-time eviction

When a restoring node replays its .rdat, it must apply the same eviction rules inline — otherwise the heap explodes during replay. Restoring nodes never write to Infinispan during replay.

10 · Recovery scenarios

ScenarioImpactRecovery
Neeve Primary fails Backup promotes instantly Seconds — automatic via Neeve HA
Neeve Backup fails No user impact Restart, replays own .rdat
Both Neeve fail TC offline; Batman/GUI still read from Infinispan (degraded) Minutes — replay .rdat from last snapshot
Infinispan node fails Other nodes hold backup partition Seconds — automatic rebalance
Infinispan cluster total failure Queries fail; Neeve still operating Rebuild from Neeve active map + .rdat replay
Solace fails Trading halts platform-wide Out of scope (platform-level incident)

Restoring a failed Primary as new Backup

Phase A — Assessment (~2 min)
Confirm new Primary stable, decide recovery path
Verify failover, check Infinispan health, assess .rdat integrity, decide replay vs snapshot transfer.
Phase B — Boot restoring node (~3 min)
Start Neeve TC standalone, NOT yet registered
Initialize JVM and Guice. Verify connectivity to Solace, Infinispan, peer TC. Do NOT subscribe to Solace yet. Do NOT write to Infinispan.
Phase C — State recovery (1-10 min)
Replay local .rdat, then catch up via Solace
Replay journal with eviction logic applied inline. Subscribe to Solace from appropriate offset. Process events through normal handlers. Continue until lag < 100ms.
Phase D — Cluster registration (~1 min)
Register with Neeve cluster as Backup
Verify mirroring active. Run optional reconciliation check against Primary. Confirm Lumino dashboard shows healthy P+B.
Phase E — Validation (~2 min)
Verify HA restored
Send synthetic order. Verify both nodes process correctly. Verify only Primary writes to Infinispan. Sign off.
Total recovery: ~15 minutes typical, up to 45 worst case. During this window the system has no redundancy — a second failure causes full TC outage. Risk acceptable given low double-failure probability and continued Infinispan availability for read-only queries.

Reconciliation on promotion

When Backup promotes to Primary, the in-flight async writes to Infinispan from old Primary may not have all landed. Promotion-time reconciliation: new Primary iterates its activeOrders map and re-writes any orders missing from Infinispan. Bounded time (seconds for ~50-200k active orders), runs once.

Failback policy

Do NOT failback automatically. Once Backup promotes, it stays Primary. The restored node becomes the new Backup. Failback only at next planned maintenance window. Every switchover is a risk; minimise unnecessary switchovers.

11 · Debuggability trade-offs

Honest assessment: pure Neeve gives perfect deterministic replay from .rdat. Adding Infinispan introduces a second source of state and async write timing — debuggability does take a step back for some bug classes.

Bug classPure Neeve todayNeeve + Infinispan
Handler logic bug Deterministic via .rdat replay Same — replay still works
Eviction logic bug n/a (no eviction today) Deterministic via .rdat replay
Query routing bug n/a Deterministic — local code
State divergence (P vs Infinispan) n/a Harder — needs cross-system reconstruction
Async write timing race n/a May not be deterministic
Infinispan-internal bug n/a Vendor support / Hot Rod tracing

Mitigations

  • Sync writes in debug environments: Guice can bind SyncInfinispanWriter in dev/UAT, AsyncInfinispanWriter in production. Removes timing variable in repro.
  • Disable Infinispan writes during replay: Replay-mode flag short-circuits Infinispan path entirely — Neeve-side bugs reproduce against just .rdat
  • Comprehensive integration logging: Every Infinispan operation logged on both sides with correlation IDs
  • Periodic Infinispan snapshots: Cache store with disk persistence enables point-in-time reconstruction
Net assessment: ~80% of bug classes remain as debuggable as today. ~20% (cross-system, async timing, Infinispan-internal) become harder but are bounded by tooling. Acceptable trade-off given memory and scale benefits.

12 · Deployment model

Infinispan Hot Rod client (in Neeve TC JVMs)

Pure Java library — no JBoss, no Wildfly, no application server required. Simple Maven dependency. Compatible with Guice DI.

<dependency>
  <groupId>org.infinispan</groupId>
  <artifactId>infinispan-client-hotrod</artifactId>
  <version>14.x.x</version>
</dependency>

Infinispan server cluster (separate)

Standalone server processes deployed on OpenShift via Red Hat Data Grid Operator. Managed by platform team. Trading team consumes as a service.

Bottom line: Replace the unbounded transaction cache pattern with a tiered architecture. Neeve TC stays as the lean write path with eviction. Red Hat Data Grid (Infinispan) absorbs the full-day query store. Batman and GUI clients see zero API change because the query interface remains message-driven over Solace. Procurement, operations, and serialization all align with existing Barclays standards (RHEL, OpenShift, protobuf, Red Hat agreements). The design is conservative, low-risk, and preserves all working components — only adding one new strategically-aligned dependency.
